# 数据展示与测试平台 · 分层架构设计

| 项 | 内容 |
|---|---|
| 状态 | 草案 v1(供评估) |
| 范围 | 把"采集 / 存储 / 展示 / 测试驱动"解耦为分层架构,支撑多硬件、数据持续增长、InferenceX 级展示 |
| 受众 | 架构评审 / 决策者 / 实施工程师 |

## 1. 背景与现状

### 1.1 已有能力(资产,不要推倒)
- **存储**:`data/benchmark.db`(SQLite),已累积 **262 runs / 32,159 results / 6 application_cases**;八维 schema(test_runs/test_results/application_cases + model_spec/serving_config/resource_monitor JSON 列)。
- **查询/导出**:`core/warehouse/`(query.py: project_run/build_cross_matrix/distinct_values;export.py:CSV/JSON/ZIP;templates.py:hwInventory/hmTest/maTest;capability_sheet;scaling_efficiency)。
- **采集模块(纯函数,已解耦)**:`core/hardware_fingerprint.py`(CPU/内存/8×GPU/CUDA/machine_id)、`core/engine_capture.py`(多引擎适配器:docker inspect + 日志 + /v1/models)、`core/resource_monitor.py`(per-GPU nvtop 级:温度/功耗/PCIe/throttle)。
- **测试驱动**:`core/benchmark_runner.py`(async,矩阵/并发/prefill/长上下文等)+ `core/historical_import.py`(历史回填)。
- **展示**:`ui/warehouse_browser.py`(7 tab:运行历史/透视矩阵/扩展效率/硬件盘点/应用用例/能力表/导出),接入 `app.py`。

### 1.2 问题(两处过深耦合)
| 耦合 | 表现 | 后果 |
|---|---|---|
| **展示 ↔ 测试平台** | warehouse_browser 是 `app.py`(测试 app)的一个 tab | 只看数据也得跑整个测试 app(conda 环境 / runner / localhost 引擎假设)→ 笔记本、给别人看、独立部署都困难 |
| **监控/采集 ↔ 本机** | ResourceMonitor 采**本机** NVML;engine_capture `docker inspect` **本机** | 测**远程/其他硬件**时,采到的是压测机的资源,不是被测硬件的;引擎配置也拿不到(远程无 docker 访问) |

### 1.3 触发
- 数据持续增长 → 需要专业、快捷、可共享的阅览。
- 要在**其他硬件**上测试(不同 GPU/机型/引擎)→ 当前架构假设"压测机==被测硬件"。
- InferenceX 参照:它本身就是**解耦的展示平台**,被多源 benchmark 数据喂养。

## 2. 设计目标与非目标

### 目标
1. **G1 解耦展示**:展示层独立运行,只读仓库,不依赖测试 runner / 本机假设。
2. **G2 多硬件**:在被测硬件上采集它的真实指纹/资源/引擎配置;支持远程硬件。
3. **G3 可扩展**:数据量增长 / 多机器 / 多引擎 / 多模型,查询与展示不退化。
4. **G4 兼容现状**:现有 DB schema、warehouse 模块、采集模块**复用**,不做大爆改;现有单机流程继续可用。
5. **G5 InferenceX 级展示**:交互对比(跨机器/引擎/模型 Pareto、并发扫描、原始数据导出)。

### 非目标(本期不做)
- 不自建分布式 DB(SQLite → 后期可换 Postgres,接口不变)。
- 不做实时流式(批次/按测试完成入库即可)。
- 不做多租户/鉴权(先单组织;预留)。

## 3. 分层架构总览

### 3.1 四层 + 组件图
```
┌─────────────────────────────────────────────────────────────────┐
│ ④ 展示层 Display(独立 app,只读仓库)                              │
│   standalone Streamlit/Web · 交互对比图 · 模板导出 · InferenceX 风 │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 只经 查询 API(REST/直连)
┌───────────────────────────┴─────────────────────────────────────┐
│ ③ 仓库层 Warehouse(中心,单一真相,硬件/引擎无关)                  │
│   SQLite/DB · repositories · query/export/templates · 摄入(ingest)│
└───^─────────────────────────────────────────────────────^───────┘
    │ 摄入 API(file/HTTP)                          摄入 │(results+probe 数据)
┌───┴──────────────────┐                       ┌──────────┴────────┐
│ ① 采集探针 Probe      │  ←可选 co-locate→    │ ② 测试驱动 Driver  │
│ (跑在被测硬件上)       │                       │ (跑哪都行,发请求)  │
│ hardware_fingerprint  │                       │ benchmark_runner  │
│ ResourceMonitor       │                       │ (httpx 负载)       │
│ engine_capture(HTTP优)│                       │ providers         │
│ → 产 probe JSON       │                       │ → 报 results       │
└───────────────────────┘                       └───────────────────┘
        跑在被测硬件                                   跑在任意机
```

### 3.2 依赖规则(硬约束,评审重点)
| 层 | 允许依赖 | 禁止依赖 |
|---|---|---|
| ④ Display | ③ 查询 API | 禁止 benchmark_runner / providers / 本机硬件 / app.py |
| ③ Warehouse | DB + 自身 query/export | 禁止 任何上游(Probe/Driver 通过 ingest API 喂数据,不反向调用) |
| ① Probe | capture 模块(fingerprint/engine_capture/ResourceMonitor)+ ingest API | 禁止 测试逻辑 / runner |
| ② Driver | providers + ingest API;**可选** co-locate Probe | 禁止 直接连 Display |

**关键不变量**:数据单向流(Probe/Driver → Warehouse → Display),**无反向依赖、无环**。Display 永远不 import runner。

### 3.3 每层一句话职责
- **Probe**:在被测硬件上,采集它的硬件指纹 + 资源时序 + 引擎配置,按协议产 JSON 入仓。
- **Driver**:对引擎端点发负载请求,产 per-request 性能结果,入仓(可附带同机 Probe 数据)。
- **Warehouse**:中心存储 + 摄入校验 + 查询/导出。硬件/引擎/测试方式无关。
- **Display**:只读仓库,交互展示与对比,可独立部署/分享。

## 4. 各层详设计

### 4.1 采集探针层 Probe
**职责**:在被测硬件上采集该硬件的真实画像,与"谁在发请求"解耦。
**采集内容**(对应八维 A/B/C/F):
- A 硬件:`capture_hardware_fingerprint()`(machine_id/CPU/内存/8×GPU/CUDA)——**一次**或按需刷新。
- B 资源:`ResourceMonitor` 时序(CPU/GPU util/vram/power/温度/PCIe/throttle)——**测试期间**持续采样。
- C 引擎:`capture_engine_config()`(**HTTP 优先**:远程走 /v1/models + /metrics;本机兜底 docker inspect + 日志)。

**两种形态**:
- **嵌入式**(短期):Driver 在被测硬件上跑时,就地调 Probe 模块(= 现状,co-located)。
- **独立式**(多硬件):`probe` CLI,在被测硬件上常驻/按需运行,产 probe JSON 包。

**新增**:`core/probe/__init__.py` 包装 fingerprint+monitor+engine_capture 为 `run_probe(duration, engine_url) -> ProbePacket`。

### 4.2 测试驱动层 Driver
**职责**:发负载、产性能结果。**硬件无关**(只认 HTTP 端点)。
- 现有 `benchmark_runner` 就是 Driver 主体;只需把"资源监控/指纹/引擎采集"从 runner 内联调用,改为**接收一个 ProbePacket**(或 co-locate 调 probe 模块)。
- Driver 不假设本机有 GPU;若需资源数据,显式绑定一个 Probe(本机或远程)。

### 4.3 仓库层 Warehouse
**复用**:`core/warehouse/` 全部、DB schema、repositories。
**新增**:
- **摄入 API**(`core/warehouse/ingest.py`):接收 ProbePacket + Driver results → 校验 → 写 DB(幂等:按 run_id + machine_id 去重)。
- **查询 API**(`core/warehouse/api.py`,薄):封装 query.py 为可被 Display 调用的接口(进程内直连 or REST)。
**不变**:schema 向后兼容(已有 model_spec/serving_config/resource_monitor JSON 列)。

### 4.4 展示层 Display
**职责**:只读仓库,交互展示。
- **短期**:把 `ui/warehouse_browser.py` 拆成**独立 Streamlit app**(`display_app.py`,只连 SQLite + import core/warehouse,不 import runner/app.py)。
- **长期**:InferenceX 风格 Web app(交互 Pareto、并发扫描曲线、跨 machine/engine/model 对比、原始数据导出、按日期/硬件筛选)。

## 5. Probe 协议(核心,详细)

### 5.1 ProbePacket JSON schema(与八维对齐,版本化)
```json
{
  "protocol": "probe.v1",
  "captured_at": "2026-06-17T10:00:00",
  "machine_id": "ab8652ab0b09bbd7",
  "hardware": {                       // A 维 = capture_hardware_fingerprint() 全量
    "cpu": {...}, "memory": {...}, "gpus": [...], "cuda": {...}, "os": {...}
  },
  "engine": {                         // C 维 = capture_engine_config() 输出
    "engine": "vLLM", "engine_version": "0.23.0",
    "launch_cmd": "...", "schedule": {"max_num_seqs": 64, ...},
    "parallel": {"tp": 8, ...}, "runtime": {"kv_cache_tokens": 1529216, ...},
    "capture_source": ["api", "docker_inspect", "docker_logs"]
  },
  "resource_timeline": [              // B 维 = ResourceMonitor 时序(可选,测试期产)
    {"t": 0.0, "cpu_percent": 8.0, "gpu_util_percent": 100.0, "gpu_temp_c": 85, "per_gpu": [...]},
    ...
  ],
  "resource_summary": {               // B 维 峰值/均值 = monitor.stop()
    "peaks": {...}, "per_gpu_peaks": [...], "throttle_reasons_seen": [...]
  },
  "test_window": {                    // 关联:这段时间的资源对应哪个 run
    "run_id_hint": "uuid", "start_ts": "...", "end_ts": "..."
  }
}
```
**字段来源标签**:hardware/engine 自动;resource 自动;test_window 由 Driver 填(关联 run)。

### 5.2 传输(两档,渐进)
- **v1 文件**(零基建,先做):Probe 产 `probe_<machine_id>_<ts>.json`,Driver/脚本 `ingest(path)` 入库;或落共享卷。**幂等**:按 `machine_id + captured_at + run_id_hint` 去重。
- **v2 HTTP**(多机):`POST /ingest/probe`(`core/warehouse/api.py` 暴露),Probe 推送。带 token(预留鉴权)。失败重试 + 本地落盘兜底(不丢数据)。

### 5.3 远程引擎采集(HTTP 优先,关键)
远程硬件上 Probe **无 docker 访问**,engine_capture 改为 HTTP 优先:
1. `/v1/models` → model_id / max_model_len(任何 OpenAI 兼容引擎)。
2. `/metrics`(Prometheus,vLLM/SGLang 暴露)→ 解析 `vllm:gpu_cache_usage_perc`、运行队列、版本等(适配器扩展)。
3. docker inspect / 日志解析 = **仅本机时**的增强(远程降级,标记 `capture_source` 缺哪些)。
→ `engine_capture` 已是多引擎适配器架构,加一个 `RemoteMode`(HTTP-only)分支即可。

### 5.4 远程 GPU 资源监控(两路)
- **优先**:读远程引擎 `/metrics`(vLLM 经 DCGM/自己暴露 GPU util/mem/power/temp)——**零部署**。
- **兜底**:远程机跑 Probe(NVML),ship 时序——**精度高**,需部署 Probe。
→ ResourceMonitor 抽 `Source` 接口:`LocalNVMLSource`(现状)/ `PrometheusMetricsSource`(新,远程)/ `RemoteProbeSource`(消费远端 Probe 时序)。

## 6. 数据流(时序)

### 6.1 单机 co-located(现状兼容,平滑过渡)
```
Driver(benchmark_runner) ──co-locate──> Probe(本机 fingerprint+monitor+engine_capture)
       │ 发请求 + 采资源                         │
       │                                        │
       └──── results + ProbePacket ───> Warehouse.ingest ─> DB
                                                              ^
Display(独立 app) ─── query API ────────────────────────────┘
```

### 6.2 远程硬件(被测硬件 ≠ 压测机)
```
[被测硬件]  Probe ── fingerprint + engine(/v1/models+/metrics) + 资源(/metrics 或 NVML)
              │
              │  ProbePacket (HTTP/file)
              v
[中心]  Warehouse.ingest ─> DB
              ^
              │  results (per-request 性能)
[压测机]  Driver ── 发请求到 被测硬件引擎端点

Display(任意机) ── query API ──> Warehouse
```
**要点**:Driver 在压测机发请求,Probe 在被测硬件采资源,两者用 `run_id` + `machine_id` 关联到同一 run。Display 完全独立。

### 6.3 多硬件汇总
```
硬件A Probe ─┐
硬件B Probe ─┼─> 中心 Warehouse ─> Display 按 machine_id/engine/model 分组对比
硬件C Probe ─┘    (Pareto: 吞吐/延迟/成本 跨硬件)
```

### 6.4 展示查询
Display → `query_api.list_runs(filter{machine,engine,model,date})` → `project_run` → 交互图。查询层已有 build_cross_matrix/scaling_efficiency/capability_sheet,Display 直接用。

## 7. 数据模型与标识

- **machine_id**(fingerprint 算,稳定哈希):跨重启稳定,多硬件天然分组键。已有。
- **run**:一次测试执行(test_runs),绑 machine_id + model_id + serving_config + 时间窗。
- **result**:per-request(test_results),绑 run_id。
- **probe_packet**:绑 machine_id + 时间窗 + run_id_hint(关联到 run)。
- 关系:`machine 1—N run 1—N result`;`machine 1—N probe_packet`;`run 1—1 probe_packet`(同窗)。
- 八维在 run 级聚合(A/B/C/F 维 JSON)或 result 级(D/E 维)——已有 schema 支持。

## 8. 接口契约

### 8.1 摄入(Probe/Driver → Warehouse)
- `ingest_probe(packet: ProbePacket) -> {accepted, run_id, deduped}`(幂等)。
- `ingest_results(run_id, results: list[Result]) -> {accepted, count}`。
- 校验:`protocol` 版本、必填字段(machine_id/captured_at)、格式;不符落 `quarantine` 不阻塞。

### 8.2 查询(Warehouse → Display)
- `list_runs(filter) -> list[RunView]`(project_run 投影)。
- `cross_matrix(row=machine_id, col=model_name, metric=decode_tps)`。
- `scaling_efficiency(machine_id, model_id)`、`capability_sheet(filter)`。
- 均已存在于 `core/warehouse/query.py`,薄封装即可。

## 10. 迁移路径(增量,低风险)

- **Phase 0(已完成)**:采集模块已解耦(fingerprint/engine_capture/monitor 纯函数);warehouse 查询/导出层完整;DB schema 八维就位。
- **Phase 1(解耦展示,1-2 天)**:把 warehouse_browser 拆成 `display_app.py` 独立 Streamlit(只连 DB + core/warehouse)→ **立刻**解决"看数据要跑测试 app"。零数据迁移。
- **Phase 2(Probe 化 + 摄入 API,3-5 天)**:`core/probe/` 包装采集为 `run_probe()`;`core/warehouse/ingest.py` 摄入 API(file 先,HTTP 后);runner 改为"接收/调用 Probe 产 ProbePacket"而非内联。
- **Phase 3(远程支持,3-5 天)**:engine_capture 加 HTTP 优先(RemoteMode);ResourceMonitor 加 PrometheusMetricsSource;验证远程硬件端到端。
- **Phase 4(InferenceX 级展示,持续)**:Display 升级交互对比(跨硬件/引擎/模型 Pareto、并发扫描、导出);可换前端(Plotly Dash / Next)。

每 Phase 独立可交付、可回退;Phase 1 即见效。

## 11. 非功能与风险

- **可靠性**:Probe/摄入失败**不得影响测试**(本批次已贯彻:采集全 try/except 优雅降级)。GPU 掉线事故教训——Probe 资源监控要能扛 NVML 不可用。
- **规模**:SQLite 单机到百万级 result 无压力;超量再换 Postgres(query 层接口不变)。时序 timeline 已降采样(≤3600 点)。
- **安全**:HTTP 摄入预留 token/TLS(本期 file 先,不暴露端口)。Display 只读,无写风险。
- **Probe 开销**:1s 采样 + NVML,均摊 <1% CPU;per-cell monitor 已验证不影响 TTFT 测量。
- **数据一致性**:幂等摄入(run_id+machine_id+ts 去重);schema 版本号(`probe.v1`)演进不破坏旧数据。

## 13. 评估要点(供决策)

1. **不是另起平台,是把已有资产分层解耦**——采集/仓库/查询/导出已就位(80% 复用),耦合只在"调用方式 + browser 塞在 app 里"。
2. **Phase 1(拆独立展示)即见效**,投入 1-2 天,解决最大痛点(看数据不用跑测试 app)。
3. **多硬件 = Probe + HTTP 优先引擎采集 + machine_id 分组**,不需要重设计 schema。
4. **依赖规则是核心约束**:数据单向流、Display 不碰 runner——评审重点确认这条。
5. **风险可控**:每 Phase 独立可交付/可回退;采集全优雅降级(已有 GPU 掉线教训内化)。
