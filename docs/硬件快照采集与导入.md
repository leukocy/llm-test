# 硬件快照采集与导入

> 不部署完整 llm-test 也能采集机器环境信息，导出单文件快照，中心机导入整合到数据仓库。
> 配合 [`环境信息采集指南.md`](环境信息采集指南.md) 阅读：本工具采集其中的 **A 维(硬件) + B 维(系统)**，
> 可选 **C 维(引擎) + D 维(模型架构)** 的「快照态」；F/G 维(per-test 资源监控 / 引擎运行时指标)
> 仍由 `live_bench.py` 在测试时采，不属于机器盘点。

---

## 为什么需要

`pip install -e .` 会装上 streamlit/pandas/transformers/datasets/fastapi 等重依赖。
对「只贡献环境信息、不跑基准」的机器，部署完整平台既慢又无意义。本工具把采集逻辑
抽成**单文件 zipapp**：scp 过去 `python hw-snapshot.pyz` 即可采，**零安装**。

采集器本就是纯 stdlib 的（`core/hardware_fingerprint.py` / `core/system_info.py`，
psutil/pynvml/torch 全部懒加载降级），只是被埋在重依赖项目里。本工具把它们连同
`core/engine_capture.py`(C 维) / `core/model_spec.py`(D 维) 打包，从 canonical 源生成，
无代码漂移——改 `core/` 后重新 `python tools/make_hw_snapshot_bundle.py` 刷新即可。

## 工作流

```
┌─────────────────────────┐        scp snap.json         ┌──────────────────────────┐
│  目标机器（裸机，零安装） │  ─────────────────────────→  │  中心机（已部署 llm-test） │
│                         │                              │                          │
│  python hw-snapshot.pyz │                              │  python import_hw_snapshot│
│    -o snap.json         │                              │    snap*.json --dedupe   │
│    [--probe-engine URL] │                              │    --export-hwInventory  │
│    [--model-config PATH]│                              │      inv.csv             │
└─────────────────────────┘                              └──────────────────────────┘
```

### 1. 采集（目标机器）

#### 零安装方式（推荐）

开发机先打包单文件 zipapp：

```bash
python tools/make_hw_snapshot_bundle.py        # → dist/hw-snapshot.pyz (~30KB)
```

拷到目标机器并运行：

```bash
scp dist/hw-snapshot.pyz targethost:
ssh targethost 'python hw-snapshot.pyz -o snap.json'
# 取回
scp targethost:snap.json .
```

#### 仓库内方式（开发机，已 pip install -e .）

```bash
python hw_snapshot.py -o snap.json
```

#### 可选：顺带采引擎 / 模型架构

机器上正在跑推理引擎时，采 C 维（docker inspect + /v1/models + 引擎适配器）：

```bash
python hw-snapshot.pyz --probe-engine http://127.0.0.1:8000/v1 -o snap.json
```

有模型 `config.json` 时，采 D 维（架构/参数量/quant/MTP）：

```bash
python hw-snapshot.pyz --model-config /data/glm5/config.json -o snap.json
```

#### 人工补字段

采集器采不到的盘点字段（负责人/机房/电源/散热/产品线）用 CLI 补，写入快照 `manual`：

```bash
python hw_snapshot.py -o snap.json \
    --owner 张三 --location 机房A-机柜3 --product-line 数据中心 \
    --psu-w 2000 --cooling 风冷 --remark 主力测试机
```

### 2. 导入整合（中心机）

```bash
# 批量导入（支持 glob），同机同指纹去重
python import_hw_snapshot.py snapshots/*.json --dedupe --db data/benchmark.db

# 导入并导出整合后的硬件盘点 CSV（复用 warehouse 模板）
python import_hw_snapshot.py snapshots/*.json --dedupe --export-hwInventory inv.csv
```

`inv.csv` 是仓库「硬件盘点」模板（一台机器一行），字段口径见
`core/warehouse/templates.py` 的 `HARDWARE_INVENTORY_FIELDS`。

## sudo 密码（采内存 DIMM 细节）

整个工具**只有 dmidecode 那 4 个内存细节字段需要 root**：`memory_type`（DDR5/HBM）、
`memory_channels_populated`、`memory_speed_mtps`、`ecc_enabled`。其余全部普通用户可跑
（lscpu / nvidia-smi / lsblk / NVML / docker 组权限）。`memory_capacity_gb` 由 psutil
拿到，`machine_id` 不依赖这 4 个字段——所以**无 root 时其余所有列正常，仅这 4 个留空**。

### 三种提权方式（按推荐顺序）

**方式 1：配 sudoers NOPASSWD（一次性，最省事）**

在每台目标机器上 root 执行一次，只授权 dmidecode 这一条命令（不是给全 root）：

```bash
echo "用户名 ALL=(root) NOPASSWD: /usr/sbin/dmidecode -t memory" \
  | sudo tee /etc/sudoers.d/dmidecode-collector
```

之后普通用户跑 `hw-snapshot.pyz`，代码自动 `sudo -n dmidecode` 免密采到。无需每次给密码。

**方式 2：给 sudo 密码（无 NOPASSWD 时）**

密码**只经 stdin 透传给 `sudo -S`，绝不进 argv/ps、不落快照**。三种安全输入方式：

```bash
# 交互输入（getpass 不回显，不进 shell history）—— 最安全
python hw-snapshot.pyz --ask-sudo-password -o snap.json

# 环境变量（CI/脚本场景；进程结束即随 env 消失）
HW_SNAPSHOT_SUDO_PASSWORD=xxx python hw-snapshot.pyz -o snap.json

# 文件（建议权限 0600，用完删除）
echo 'xxx' > /tmp/pw && chmod 600 /tmp/pw
python hw-snapshot.pyz --sudo-password-file /tmp/pw -o snap.json
rm /tmp/pw
```

> ⚠️ **没有 `--sudo-password <值>` 参数**——那会进 `ps aux` / shell history，不安全。
> 代码主动拒绝这种写法。

**方式 3：直接 root 跑** `sudo python hw-snapshot.pyz`（产出文件 owner 是 root，取回时 chown）。

### 安全保证（已实测 + 测试守卫）

| 风险 | 处理 | 验证 |
|---|---|---|
| 密码进 `ps aux`/进程列表 | `sudo -S` 从 stdin 读，argv 只有 `sudo -S -k dmidecode -t memory` | `test_password_never_in_argv` |
| 密码进 shell history | 无 `--sudo-password` 值参数；交互用 getpass | — |
| 密码落快照 JSON | `sudo_password` 绝不写入快照；`build_snapshot` 文档明确声明 | `test_password_not_in_snapshot_json` |
| 密码残留内存 | 用完即 `del`，进程结束释放 | CLI `del sudo_password` |
| 权限范围 | sudoers 可限定 `NOPASSWD: /usr/sbin/dmidecode -t memory`，仅此一条命令 | — |

降级链：`dmidecode`(无权限) → `sudo -n`(NOPASSWD) → `sudo -S`(密码) → 留空。
`sudo -n` 成功时不再走 `sudo -S`（不浪费密码）。无密码且 `sudo -n` 也失败时，4 个字段留空，
其余不受影响。

### 实际使用流程（两种典型场景）

#### 场景 A：自己的固定机器 —— 配一次 sudoers，永久免密

适合你长期管理、能 root 的机器。配一次之后每次采集不用再碰密码。

```bash
# 中心机：打包采集器（一次性）
python tools/make_hw_snapshot_bundle.py        # → dist/hw-snapshot.pyz

# ── 在每台目标机器上执行一次（root）──
# 只授权 dmidecode 这一条命令，不是给全 root
sudo bash -c 'echo "$(whoami) ALL=(root) NOPASSWD: /usr/sbin/dmidecode -t memory" \
  > /etc/sudoers.d/dmidecode-collector && chmod 440 /etc/sudoers.d/dmidecode-collector'

# ── 之后每次采集（普通用户，无需任何密码）──
scp dist/hw-snapshot.pyz 目标机器:~/
ssh 目标机器 'python hw-snapshot.pyz -o snap.json'
scp 目标机器:~/snap.json ./snap_目标机器.json
```

代码自动 `sudo -n dmidecode` 免密采到内存细节（DDR5/通道/频率/ECC）。

#### 场景 B：临时去别人机器 —— 交互输入密码，不回显

适合临时采集、不能改对方 sudoers 的机器。密码用 `getpass` 读取，不回显、不进 history、不落快照。

```bash
# 中心机：打包（一次性）
python tools/make_hw_snapshot_bundle.py

# ── 采集（交互输入 sudo 密码）──
scp dist/hw-snapshot.pyz 目标机器:~/
ssh -t 目标机器 'python hw-snapshot.pyz --ask-sudo-password -o snap.json'
#   提示：sudo 密码（用于 dmidecode 采内存细节，不回显）: ████  ← 输入，看不见
scp 目标机器:~/snap.json ./snap_目标机器.json
```

> `ssh -t` 强制分配伪终端，`getpass` 才能正常隐藏输入；不加 `-t` 密码会回显。

#### 两种场景对比

| | 场景 A（sudoers） | 场景 B（交互密码） |
|---|---|---|
| 适用 | 自己的固定机器 | 临时去别人机器 |
| 配置 | 一次性配 sudoers | 无需配置 |
| 每次操作 | 直接跑 | 每次输一次密码 |
| 密码暴露面 | 无密码 | getpass 不回显，不进 history/ps/快照 |
| 权限范围 | 仅 `dmidecode -t memory` 一条 | 仅 `dmidecode -t memory` 一条 |
| 采到的内存字段 | 完全相同（DDR5/通道/频率/ECC） | 完全相同 |

两种方式采到的内存字段完全一样；其它字段（CPU/GPU/CUDA/磁盘）普通用户就能采，与提权方式无关。

#### 中心机：导入整合（两种场景通用）

```bash
# 批量导入各机器采回的快照，同机同指纹去重，导出硬件盘点 CSV
python import_hw_snapshot.py snap_*.json --dedupe --export-hwInventory 硬件盘点.csv
```

## 快照格式（hw-snapshot/v1）

单文件 JSON，schema 版本化（导入端校验，不兼容时拒绝）：

```jsonc
{
  "schema": "hw-snapshot/v1",
  "collector_version": "1.x.x",
  "collected_at": "2026-06-24T01:00:00",
  "machine_id": "a95299ef02e04e29",          // 跨重启稳定，hwInventory 去重键
  "hostname": "heyi",
  "hardware_fingerprint": { ... },            // A 维（与 system_info.hardware_fingerprint 同构）
  "system_info": { ... },                     // B 维，内嵌 hardware_fingerprint + machine_id
  "engine_capture": { "engine": "vllm", ... },// C 维（可选，--probe-engine）
  "model_spec": { "name": "glm-5", ... },     // D 维（可选，--model-config）
  "manual": { "owner": "张三", "location": "机房A" }  // 人工补字段
}
```

## 导入后的数据归属

快照转成一条 `TestRun` 写入 `test_runs` 表（**不新建表**，零 schema 迁移）：

| TestRun 字段 | 快照来源 |
|---|---|
| `test_type` / `status_detail` | `"hardware_inventory"`（导入端标识） |
| `system_info_json` | A+B 维（hardware_fingerprint + system_info） |
| `serving_config_json` | C 维 engine_capture（可选） |
| `model_spec_json` | D 维 model_spec（可选） |
| `machine_id`（一等列） | 快照 machine_id（hwInventory 去重键） |
| `tester` | manual.owner |
| `notes` | manual.remark |
| `config_json` | manual 人工列（product_line/location/电源/散热/SSD） |
| `external_level` | `"internal"`（机器盘点不直接对外） |

**磁盘兜底**：采集器采了 `disks[]`(lsblk)，但 `project_run()` 的 `ssd_model`/`ssd_capacity_tb`
原本只读 `config`。导入时若 manual 未给 SSD，从 `disks[]` 取最大 SSD 填进 `config`；
`project_run()` 也加了同款兜底，让所有来源的磁盘数据都能在 hwInventory 导出里浮现。

**hmTest 守卫**：`build_hm_test_rows()` 跳过 `status_detail=='hardware_inventory'` 的行
（硬件盘点不是「硬件×模型测试」，不污染测试矩阵）；`build_hardware_inventory_rows()` 按
`machine_id` 收纳去重取最新。

## 去重

`--dedupe`：同 `machine_id` 且**硬件指纹哈希**一致 → 跳过。指纹哈希只纳入硬件本体属性
（CPU/内存/GPU/磁盘），忽略时间戳/可用内存/引擎态等易变值——否则机器重启一次就判定为
「新机器」会无限增长。GPU 数变化等真实硬件变更会改变哈希，正常导入新行。
