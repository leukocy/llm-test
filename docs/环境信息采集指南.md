# 环境信息采集指南

> 每个指标的采集方式、采集阶段、数据来源、持久化位置。
> 配合 `standards/测试数据字段标准.md`(定义"有哪些字段")阅读;本文定义"怎么采、何时采"。

## 采集阶段总览

```
阶段 0  sidebar 手动填写(测试前,可选)
        ↓ 用户点击"开始测试"
阶段 1  测试启动时一次性采集(_start_db_run)
        ↓ 测试运行中
阶段 2  测试运行中持续采样(后台线程)
        ↓ 测试结束
阶段 3  测试结束时汇总持久化(_complete_db_run)
```

## A 维 · 硬件

全部**阶段 1 一次性采集**,全自动,零手动。

| 指标 | 采集函数 | 数据源 | 采集方式 | 持久化 |
|---|---|---|---|---|
| CPU 型号 | `_query_cpu_topology()` | `lscpu`(LC_ALL=C) + `/proc/cpuinfo` 交叉验证 | 子进程解析,psutil 兜底 | `system_info_json.hardware_fingerprint.cpu.model_name` |
| CPU sockets/核/线程 | 同上 | `lscpu` 的 `Socket(s)`/`Core(s) per socket`/`Thread(s) per core` | 解析 lscpu 输出 | `...cpu.sockets/cores_per_socket/threads_per_core` |
| NUMA 节点数 | 同上 | `lscpu` 的 `NUMA node(s)` | 解析 | `...cpu.numa_nodes` |
| 内存类型/容量 | `_query_memory_details()` | `psutil.virtual_memory()` | Python API | `...memory.total_gb` |
| 内存通道/频率/ECC | 同上 | `dmidecode -t memory`(需 root) | 子进程解析 DIMM 条目 | `...memory.channels/speed_mt_s/ecc` |
| GPU 型号/显存 | `_query_gpus()` | `nvidia-smi --query-gpu=index,name,memory.total` | 子进程解析;torch.cuda 兜底 | `...gpus[].name/vram_gb` |
| GPU PCIe Gen×宽 | 同上 | `nvidia-smi --query-gpu=pcie.link.gen.max,pcie.link.width.max` | 子进程解析 | `...gpus[].pcie_gen/pcie_width` |
| GPU 标称带宽 | `_lookup_gpu_bandwidth()` | 内置查表(GPU 名 → GB/s) | 字典查表;查不到退化为 PCIe 公式 | `...gpus[].nominal_bandwidth_gbps` |
| GPU 互联拓扑 | `_query_gpu_topology()` | `nvidia-smi topo -m` | 子进程解析矩阵 + Legend | `...gpu_topology.has_nvlink/matrix[]` |
| CUDA/驱动版本 | `_query_cuda_versions()` | NVML(`pynvml.nvmlSystemGetDriverVersion/CudaDriverVersion`);`nvidia-smi` 兜底 | NVML API | `...cuda.driver/cuda_version` |
| 磁盘型号/容量/SSD | `_query_disk_info()` | `lsblk -d -b --json` | 子进程解析 JSON | `...disks[].name/model/size_tb/is_ssd` |
| machine_id | `compute_machine_id()` | 上述 CPU+内存+GPU 字段组合 | SHA1 前 16 位 | `...machine_id` + DB 一等列 `machine_id` |

**降级**:每个子函数独立 try/except,任一失败 → 空值,不阻塞其他采集或测试。

## C 维 · 引擎/服务配置

**阶段 1 一次性采集**,自动为主,sidebar 手动为辅(merge 时自动覆盖手动的空值)。

### 采集路径(三级降级)

| 优先级 | 路径 | 触发条件 | 采集方式 |
|---|---|---|---|
| 1 | Docker 容器 | `find_vllm_container` 找到容器(端口映射或 host 网络 `--port` 匹配) | `docker inspect`(image/Cmd/Env) + `docker logs`(适配器解析) + `docker exec`(torch/cuda 版本) |
| 2 | 裸进程 | 无容器,但端口有进程监听 | `/proc/<pid>/cmdline`(launch_cmd) + `/proc/<pid>/environ`(env) + `importlib.metadata`(本地包版本) |
| 3 | 仅 API | 无容器无进程(远程 API) | 只有 `/v1/models` 的 model_id/max_model_len |

### 各指标采集细节

| 指标 | 采集函数 | 数据源 | 采集方式 | 持久化 |
|---|---|---|---|---|
| 引擎名 | `detect_adapter()` | docker image/cmd 或 `/proc/<pid>/cmdline` | 关键词匹配(vllm/sglang/llama.cpp/ktransformers/fastllm) | `serving_config_json.engine` |
| 引擎版本 | image tag 正则 / `docker exec` / `importlib.metadata` | `re.search(r":v?([\d.]+)", image)` 或容器内 `pip show` | 三级:镜像 tag → docker exec → 本地 pip | `serving_config_json.engine_version` |

| TP/DCP/EP | `VLLMAdapter.parse_logs()` | `docker logs` 中的 `non-default args: {...}` | 正则提取 `'tensor_parallel_size': 8` 等 | `serving_config_json.parallel_strategy` |

## D 维 · 模型架构

**阶段 1 一次性采集**,注册表优先 + config.json 补充。

| 指标 | 采集函数 | 数据源 | 采集方式 | 持久化 |
|---|---|---|---|---|
| 架构(dense/MoE) | `from_local_config()` | `config.json` 的 `num_experts`/`n_routed_experts`/`moe_intermediate_size`/架构名 | 字段存在 → MoE | `model_spec_json.architecture` |
| 层数/hidden/heads | 同上 | `config.json` 的 `num_hidden_layers`/`hidden_size`/`num_attention_heads` | 直接读 | `model_spec_json.num_layers/hidden_size/num_attention_heads` |
| 专家数/top_k/shared | 同上 | `config.json` 的 `n_routed_experts`/`num_experts_per_tok`/`n_shared_experts` | 直接读 | `model_spec_json.num_experts/num_experts_per_tok/num_shared_experts` |
| head_dim | 同上 + `_mla_head_dim()` | `config.json` 的 `head_dim`;MLA 无顶层 → `qk_nope_head_dim + qk_rope_head_dim` | 直接读 / 合成 | `model_spec_json.head_dim` |
| attention 类型 | `_detect_attention_type()` | `config.json` 的 `architectures`/`model_type`/`qk_nope_head_dim`/`num_key_value_heads` | DSA → MLA → Lightning → GQA/MHA 优先级链 | `model_spec_json.attention_type` |
| 权重精度/量化方法 | `_read_quant_config()` | `hf_quant_config.json` 或 `config.json` 内嵌 `quantization_config` | ModelOpt/compressed-tensors 格式解析 | `model_spec_json.weight_dtype/quant_method/group_size` |
| KV 精度 | 同上 | `quantization_config.kv_cache_scheme` | 解析 num_bits + type | `model_spec_json.kv_dtype` |
| 多模态 | `_detect_multimodal()` | `config.json` 的 `vision_config`/`multi_modal_projector`/架构名 | 字段存在 → True | `model_spec_json.is_multimodal/modalities` |
| MTP | `_detect_mtp()` | `config.json` 的 `num_nextn_predict_layers`/`speculator_config` | 字段 > 0 → True | `model_spec_json.supports_mtp/mtp_depth/mtp_method` |
| total/active params | `resolve_spec()`(注册表权威) / `from_local_config()`(估算) | 注册表硬编码 / config.json 字段推算 | 注册表优先,local 补空字段 | `model_spec_json.total_params_b/active_params_b` |
| 上下文上限 | `from_local_config()` | `config.json` 的 `max_position_embeddings` | 直接读 | `model_spec_json.max_position_embeddings` |

**注册表优先逻辑**:`resolve_spec(model_id)` 先查 `MODEL_SPEC_REGISTRY`(权威 total/active params),`from_local_config(config.json)` 补充注册表没有的字段(quant/attention_type/moe_intermediate 等),不覆盖注册表权威值。

## E 维 · 运行时资源监控

**阶段 2 持续采样**(后台 daemon 线程,每秒一次),**阶段 3 汇总**。

### 采集机制

`ResourceMonitor` 启动一个 daemon 线程,每 `interval`(默认 1.0s)调用 `_sample_once()`:

- GPU:NVML(`pynvml`)API,每张卡独立采样
- CPU/内存:`psutil` API
- 采样存入 `self._samples` 列表(内存)
- `stop()` 时调 `_summarize()` 汇总

### 各指标采样细节

| 指标 | 采集频率 | NVML/psutil API | 汇总方式 | 持久化 |
|---|---|---|---|---|
| GPU 利用率 % | 每秒/卡 | `nvmlDeviceGetUtilizationRates(h).gpu` | 峰值(max) + 均值(mean) | `resource_monitor_json.peaks.gpu_util_percent` + DB 一等列 `gpu_util_peak_pct` |
| GPU 显存 GB | 每秒/卡 | `nvmlDeviceGetMemoryInfo(h).used` | 峰值(sum 跨卡) | `...peaks.gpu_vram_gb` + DB `gpu_vram_peak_gb` |
| GPU 功耗 W | 每秒/卡 | `nvmlDeviceGetPowerUsage(h)/1000` | 峰值(sum) | `...peaks.gpu_power_w` + DB `gpu_power_peak_w` |
| GPU 温度 ℃ | 每秒/卡 | `nvmlDeviceGetTemperature(h, GPU)` | 峰值(max) | `...peaks.gpu_temp_c` + DB `gpu_temp_peak_c` |
| GPU SM/mem 时钟 MHz | 每秒/卡 | `nvmlDeviceGetClockInfo(h, SM/MEM)` | per-GPU 峰值 | `...per_gpu_peaks[].sm_clock_mhz/mem_clock_mhz` |
| GPU 风扇 % | 每秒/卡 | `nvmlDeviceGetFanSpeed(h)` | per-GPU 峰值 | `...per_gpu_peaks[].fan_pct` |
| PCIe RX/TX MB/s | 每秒/卡 | `nvmlDeviceGetPcieThroughput(h, RX/TX)` | per-GPU 峰值 | `...per_gpu_peaks[].pcie_rx_mbs/pcie_tx_mbs` |
| NVLink 吞吐 | 每秒/卡 | `nvmlDeviceGetNvLinkUtilizationCounter(h, link, dir)` | per-GPU | `...per_gpu_peaks[].nvlink_throughput` |
| Throttle 原因 | 每秒/卡 | `nvmlDeviceGetCurrentClocksThrottleReasons(h)` bitmask → 解码 | per-GPU 去重 + 全局去重 | `...per_gpu_peaks[].throttle` + `...throttle_reasons_seen` |
| GPU 功耗上限/最大时钟 | **一次性**(start 时) | `nvmlDeviceGetPowerManagementLimit(h)` / `nvmlDeviceGetMaxClockInfo(h, SM/MEM)` | 不汇总(静态) | `...gpu_static_info[].power_limit_w/max_sm_clock_mhz/max_mem_clock_mhz` |
| CPU 利用率 % | 每秒 | `psutil.cpu_percent(interval=None)` | 峰值 + 均值 | `...peaks.cpu_percent` + DB `cpu_peak_pct` |
| 系统内存 GB | 每秒 | `psutil.virtual_memory().used` | 峰值 | `...peaks.system_memory_gb` + DB `system_memory_peak_gb` |
| 时序(timeline) | 每秒 | 上述全部聚合字段 | 降采样(≤3600 点) | `...timeline[]` |

**降级**:无 NVML/pynvml → `_nvml_ok=False`,只采 CPU/内存;无 psutil → 全空,不崩。

## F 维 · 引擎运行时指标

**阶段 2 持续轮询**(后台 daemon 线程,每秒一次),**阶段 3 汇总**。

### 采集机制

`EngineMetricsPoller` 启动 daemon 线程,每 `interval`(1.0s)对引擎 `/metrics` 端点发 HTTP GET:

- 解析 Prometheus exposition 文本(`parse_prometheus`)
- 按指标前缀分派 vLLM/SGLang 提取器
- 首次成功时冻结 `cache_config`(运行期不变)
- `stop()` 时调 `_summarize()` 汇总

### 各指标采集细节

| 指标 | 采集频率 | Prometheus 指标名 | 汇总方式 | 持久化 |
|---|---|---|---|---|
| KV cache 占用率 | 每秒 | `vllm:gpu_cache_usage_perc` / `sglang:token_usage` | 峰值(max) | `engine_metrics_json.peaks.gpu_cache_usage_perc` + DB `gpu_kv_cache_usage_peak_pct` |
| 运行队列 | 每秒 | `vllm:num_requests_running` / `sglang:num_running_reqs` | 峰值 | `...peaks.num_requests_running` + DB `engine_running_requests_peak` |
| 等待队列 | 每秒 | `vllm:num_requests_waiting` / `sglang:num_queue_req` | 峰值 | `...peaks.num_requests_waiting` |
| 抢占数 | 每秒(counter) | `vllm:num_preemption` | **增量**(末值 - 首值) | `...preemption_total` + DB `num_preemption_total` |
| TTFT 均值 | 每秒(直方图) | `vllm:time_to_first_token_seconds` histogram | **末点**(累积直方图,末点 = 整体) | `...engine_means.ttft_s` |
| TPOT 均值 | 每秒(直方图) | `vllm:time_per_output_token_seconds` histogram | 末点 | `...engine_means.tpot_s` |
| 推测解码接受率 | 每秒 | `vllm:spec_token_total_acceptance_rate` / `sglang:spec_accept_rate` | 末点 | `...engine_means.spec_token_acceptance_rate` |
| prefix cache 命中率 | 每秒 | `vllm:gpu_prefix_cache_hits_total / queries_total` 比值 / `sglang:cache_hit_rate` | 末点 | `...engine_means.gpu_prefix_cache_hit_rate` |
| KV 容量 | 首次冻结 | `vllm:cache_config_info` gauge(block_size×num_gpu_blocks) | 静态 | `...cache_config.kv_capacity_tokens` + DB `kv_cache_capacity_tokens` |
| 时序(timeline) | 每秒 | 上述全部 | 降采样(≤3600 点) | `...timeline[]` |

**降级**:无 `/metrics` 端点 → 不启动;轮询 404 → 跳过;无 httpx → 跳过。不影响测试。

## G 维 · 请求级指标(测试核心)

**阶段 2 每请求采集**(非后台线程,随请求流式返回)。

| 指标 | 采集方式 | 计算函数 | 持久化 |
|---|---|---|---|
| TTFT | 流式首个 token 的 `time.monotonic()` - 请求开始时间 | `calculate_request_metrics()` | CSV `ttft` / DB `avg_ttft` |
| TPS(decode 速度) | `completion_tokens / generation_time`(减 latency_offset,可选跳首 token) | 同上 | CSV `tps` |
| TPOT | `generation_time / (completion_tokens - 1)` | 同上 | CSV `tpot_ms` |
| TPOT P95/P99 | 逐 token 间隔的 95/99 百分位(`np.percentile`) | 同上 | CSV `tpot_p95_ms/tpot_p99_ms` |
| prefill_tps | `uncached_prompt_tokens / TTFT`(剔除 cache_hit) | `one_req()` | CSV `prefill_tps` |
| cache_hit_tokens | `usage_info.prompt_tokens_details.cached_tokens` | `extract_cache_hit()` | CSV `cache_hit_tokens` |
| 批次聚合吞吐 | `total_output_tokens / decode_duration`(批次级,非单请求) | `_batch_aggregate()` | CSV `sys_output_throughput/sys_throughput/rps` |
| ITL(逐 token 间隔) | `token_timestamps[j+1] - token_timestamps[j]` | `itl_stats()` | CSV `peak_itl_ms/steady_itl_ms/converge_token` |

## 降级保证

每个采集子模块独立 try/except,任一失败不阻塞其他采集或测试:

| 场景 | 降级行为 |
|---|---|
| 无 GPU / 无 NVML | ResourceMonitor 只采 CPU/内存;hardware_fingerprint.gpus=[] |
| 无 Docker / 远程 API | engine_capture 返回 `no_container`,只有 `/v1/models`;裸进程回退 `/proc` |
| 引擎无 `/metrics` | EngineMetricsPoller 不启动;engine_metrics_json 为空 |
| 无 config.json | model_spec 用注册表;注册表也无 → 空字典 |
| 无 dmidecode 权限 | memory 只有 psutil 的总量/可用,无通道/频率/ECC |
| `--no-auto-capture` | live_bench 跳过全部自动采集,只用 `--kv`/`--engine-json` 手填 |
| 自适应探测失败 | KV 预算=None → 不跳过任何 cell,全跑(与无自适应时行为一致) |

## 自适应测试(KV 预算驱动 cell 跳过)

> 采集到的 KV 容量不只用于记录——它直接驱动测试行为:超预算的 cell 自动跳过,
> 不再跑到 OOM 才失败。这是「采集的上下文和并发限制调整测试」的兑现。

### 触发时机

**阶段 1(测试启动时)**,`_start_db_run` 调 `_probe_kv_budget()` 一次性探测 KV 预算,
写入 `self._kv_budget`。整个测试期间用这一个值判断每个 cell(永不抛异常,探测失败回退全跑)。

### KV 预算来源(优先级降级)

| 优先级 | 来源 | 采集函数 | 说明 |
|---|---|---|---|
| 1 | 手动指定 | UI 侧边栏「KV 预算」/ `warehouse_context.engine_runtime.kv_budget` | 用户最清楚时直接填 |
| 2 | 引擎 `/metrics` | `probe_kv_capacity()` → `vllm:cache_config_info`(`block_size×num_gpu_blocks`) | 最准,反映引擎实际分配的 KV 池 |
| 3 | `/v1/models` | `probe_kv_capacity()` → `max_model_len` | 模型上下文上限作保守上界(实际池可能更小) |
| 4 | 都拿不到 | `_kv_budget=None` | 不跳过任何 cell,全跑(与原行为一致) |

`probe_kv_capacity(api_base)` 是纯函数(不启动轮询线程),与 F 维的 `EngineMetricsPoller`
同源(复用 `parse_prometheus`/`extract_vllm_runtime`),但只探测一次。

### cell 跳过判断

```
need = concurrency × (context_tokens + max_tokens)
if need > kv_budget: 跳过该 cell,error 列记 "over_kv_budget (need=X > budget=Y, source=...)"
```

口径与 `live_bench.build_phases` 完全一致(`conc*ctx + conc*max_tokens ≤ kv_budget`)。

### 接入的测试方法

| 测试方法 | cell 维度 | 跳过位置 |
|---|---|---|
| `run_concurrency_test` | 并发 × `input_tokens_target` × `max_tokens` | 每个 (并发, 轮) cell 跑前 |
| `run_throughput_matrix_test` | 并发 × `context_length` × `max_tokens` | 每个 (并发, 上下文) cell 跑前 |

跳过的 cell **不静默消失**——写入 `conc=8, ctx=65536, error="over_kv_budget(...)"` 一行,
在结果 CSV / 仓库里留痕(手册:「缺测本身就是决策信息」,超预算也是决策信息)。

### UI 控制

侧边栏「🔌 推理引擎接入」面板新增「KV 预算(tokens,自适应跳过)」输入:

- **留空/0** = 测试前自动从引擎 `/metrics` 或 `/v1/models` 探测
- **填了** = 用手动值(优先级最高),适合知道确切 KV 池大小或要复现某次配置时

### 与 live_bench 的关系

`live_bench.py`(命令行脚本)早就有 `build_phases`/`--kv-budget` 自适应,但 KV 容量要手动给。
本次把同一逻辑回流到 `BenchmarkRunner`(UI 用的核心编排器),并加了自动探测——
UI 和命令行现在都能根据 KV 容量跳过超预算 cell,口径一致。
